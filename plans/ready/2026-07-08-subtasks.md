# Subtasks

**Date**: 2026-07-08
**Status**: Ready

## Problem

Todos are currently flat. Real work is often hierarchical — "Plan a birthday
party" is really a handful of smaller steps. We want **subtasks**: todos that
belong to a parent todo, synced across web, iOS, and the API, and fitting the
app's existing "describe it, AI handles the rest" ethos (a subtask is a
first-class todo, so it inherits URLs, AI enrichment, research, and sync for
free).

## Decisions (agreed during brainstorm)

1. **One level only.** Parent → subtasks, no grandchildren. A subtask cannot
   itself have subtasks.
2. **A subtask _is_ a todo.** Modelled as a self-referential `parent_id` on
   `todos`, not a separate lightweight table. Subtasks reuse the whole todo
   pipeline (notes, due date, priority, URLs, AI, research, sync).
3. **Completion is a downward master switch.**
   - Checking a parent completes **all** its subtasks; unchecking a parent
     reopens **all** its subtasks.
   - Completing every subtask does **not** auto-complete the parent (no upward
     propagation). The parent's completion is always an explicit user action.
   - Completed subtasks sink to the **bottom** of the parent's subtask list.
4. **Subtasks are permanently bound to their parent.** No promotion to
   top-level, no moving between parents. `parent_id` is immutable after
   creation. This significantly simplifies sync — there are no reparenting
   conflicts to resolve.
5. **Ordering.** Active subtasks are reorderable via fractional index within
   their sibling group. Completed subtasks are pinned to the bottom and are
   **not** reorderable.
6. **Lists.** Subtasks inherit their parent's list membership. They never
   appear independently in lists (no rows in `todo_lists` for subtasks; list
   filtering resolves through the parent).
7. **Recurrence is top-level only, mutually exclusive with subtasks.** A
   subtask cannot recur, and a todo may have subtasks **or** a recurrence, not
   both. Enforced server-side (source of truth) and mirrored in the UI.
8. **Deletes cascade; no new machinery needed.** Deleting a parent deletes its
   children. Both clients already reconcile deletions by absence (iOS always
   pulls with `lastSyncedAt: nil`; web refetches the full list), so D1's
   `ON DELETE cascade` propagates a parent deletion to its children on every
   device with no server changes. `deletedIds`/tombstones remain a worthwhile
   _future_ optimisation (true incremental delta sync) but are out of scope.
9. **Subtasks live inside the todo.** They are displayed and managed as a list
   *within* the todo's expanded/edit view (web `TodoItemExpanded`, iOS
   `TodoEditSheet`) — an "add subtask" input plus a reorderable list of child
   rows with checkboxes, completed rows pinned to the bottom. Subtasks are
   **not** rendered as nested rows in the main list; the main-list parent row
   shows only an `n/m` progress indicator.
10. **Badge/counts are top-level only.** The app icon badge and any "N todos"
    counts include only top-level todos, never subtasks, so a task with many
    subtasks never inflates the number.

## Solution

Add a nullable, self-referential `parent_id` to `todos`
(`references(() => todos.id, { onDelete: "cascade" })`). Top-level todos have
`parent_id IS NULL`; subtasks point at their parent. Positioning stays
fractional-index but is scoped to the sibling group `(user_id, parent_id)`.

The API keeps returning a **flat** list of todos (both `getTodos` and
`/todos/sync` already do this); each client reconstructs the one-level tree
from `parentId`. This keeps the API surface tiny.

## Implementation

### Phase 1 — Data model & shared schema

`src/shared/src/schema.ts`

- Add to the `todos` table:
  ```ts
  parentId: text("parent_id").references(
    (): AnySQLiteColumn => todos.id,
    { onDelete: "cascade" },
  ),
  ```
  (self-reference needs the `AnySQLiteColumn` return annotation; import from
  `drizzle-orm/sqlite-core`).
- Add an index for sibling lookups/ordering, scoped per user to match the
  `(user_id, parent_id)` sibling grouping (userId first so subtask queries stay
  per-user and don't scan across the whole table):
  ```ts
  index("idx_todos_user_parent_position").on(table.userId, table.parentId, table.position),
  ```
  Keep `idx_todos_user_position` for the top-level list.
- Extend `todosRelations` with a self relation:
  ```ts
  parent: one(todos, { fields: [todos.parentId], references: [todos.id], relationName: "subtasks" }),
  subtasks: many(todos, { relationName: "subtasks" }),
  ```

Generate the migration: `pnpm db:generate` → produces
`src/api/migrations/0017_*.sql` (`ALTER TABLE todos ADD parent_id ...` +
`CREATE INDEX idx_todos_user_parent_position`). Update
`migrations/meta/_journal.json` is handled by drizzle-kit. Apply locally with
`pnpm db:migrate`.

### Phase 2 — API

`src/api/src/handlers/sync.ts`

- **`serializeTodo`**: add `parentId: todo.parentId?.toLowerCase() ?? null`.
- **`syncRequestSchema`**: add `parentId: z.string().uuid().nullable().optional()`
  on each change.
- **Insert path**: persist `parentId` on create (normalise to lowercase).
- **Update path**: **ignore** `parentId` on update — it is immutable (do not
  read it into the `.set(...)`). Reparenting is not a supported operation.
- **Completion cascade (parent → children)**: when an update sets a top-level
  todo's `completed` to a new value, cascade it to its children in the same
  request:
  ```ts
  if (change.completed !== undefined && change.completed !== existing.completed && !existing.parentId) {
    await db.update(todos)
      .set({ completed: change.completed, completedAt: change.completed ? now : null, updatedAt: now })
      .where(eq(todos.parentId, normalizedId));
  }
  ```
  Bumping children's `updatedAt` guarantees they're picked up by the next pull
  on every client. (Guard against combining with the recurring-completion
  branch — a todo with subtasks is not allowed to recur.)
- **Delete path**: no change needed. D1 `ON DELETE cascade` removes children;
  clients reconcile by absence.
- **Recurrence × subtasks invariant (server-side)**: enforce it here, not just
  in the UI, so older clients and direct API calls can't create an illegal
  state. Reject (or drop) a `recurrence` on a todo that has children, reject a
  `parentId` create under a recurring parent, and reject adding a subtask to a
  recurring todo. Mirror the same guard in the web server functions below.

`src/web/src/server/todos.ts`

- `getTodos` already returns a flat list — add `parentId` to whatever
  serialisation/mapping the web server function returns (mirror the sync
  serializer). No ordering change required; the client groups by `parentId`.
- `createTodo` / `updateTodo`: accept and persist `parentId` on create; ignore
  on update. Add the same parent→children completion cascade on update, and the
  same server-side recurrence × subtasks guard as `sync.ts`.
- `deleteTodo`: unchanged (cascade handles children).

`src/api/src/handlers/smart-create.ts`

- No change in Phase 2. (AI subtask generation is Phase 5.) A subtask created
  manually just posts a normal todo with `parentId` set and a position
  generated within the sibling group.

### Phase 3 — Web UI

The subtask list lives **inside** the expanded todo form
(`src/web/src/components/TodoItemExpanded.tsx` — the Title / Notes / Due date /
Priority / Repeat panel), not as nested rows in the main list.

`src/web/src/components/TodoItemExpanded.tsx`

- Add a **Subtasks** section (below Notes, above or replacing the Repeat
  control per the exclusivity rule):
  - A list of child rows: checkbox + title, active rows drag-reorderable,
    completed rows pinned to the bottom and not draggable.
  - An "Add subtask" inline input that creates a todo with `parentId = todo.id`
    and a position at the end of the active sibling group.
  - Ordering: active subtasks by `position` asc, then completed subtasks below.
  - Editing a subtask's title inline is enough for v1 (no nested expand — one
    level only, so a subtask has no due date / repeat / subtasks UI here unless
    we choose to surface them later).
- **Recurrence exclusivity**: if the todo has subtasks, hide/disable the Repeat
  control; if it repeats, hide the Subtasks section (and the add-subtask input).
- Follow `make-interfaces-feel-better` for add/remove/reorder animation.

`src/web/src/components/TodoList.tsx` (+ `TodoItem*`)

- Exclude subtasks (`parentId != null`) from the main list — only top-level
  todos render as rows.
- Compute per-parent progress from the flat list and show an `n/m` indicator on
  parent rows (a small badge / progress hint).
- **Checkbox cascade**: toggling a parent's checkbox optimistically toggles all
  its subtasks (mirrors the server cascade); toggling a subtask never toggles
  the parent.
- Deleting a parent removes its subtasks (server cascade + reconcile); reflect
  optimistically by dropping children from the cache too.

### Phase 4 — iOS

> All paths below are under `src/ios/Nylon Impossible/Nylon Impossible/`.
> Note: unlike `TodoMessage`/`TodoUrl`, there is **no** `TodoItem+APIConversion.swift`.
> The `APITodo`/`TodoChange` DTOs live in `Services/APIService.swift` and the
> `TodoItem` ↔ DTO mapping is done inline in `Services/SyncService.swift`
> (`applySync` for inbound, `gatherLocalChanges` for outbound).

`Models/TodoItem.swift`

- Add `var parentId: UUID?` (immutable in practice).
- Add a self-referential SwiftData relationship for local cascade + grouping:
  ```swift
  @Relationship(deleteRule: .cascade, inverse: \TodoItem.parent) var subtasks: [TodoItem] = []
  @Relationship var parent: TodoItem?
  ```
  (Local `.cascade` keeps SwiftData consistent when a parent is deleted before
  the next server reconcile.)

`Services/APIService.swift`

- Add `parentId: String?` to the `APITodo` struct (line ~54) and to the
  `TodoChange` struct (line ~210), plus their initializers/`Codable` keys.

`Services/SyncService.swift`

- `applySync`: set `parentId` on each `TodoItem` materialised from `APITodo`.
  Existing reconcile-by-absence already deletes cascade-removed children —
  verify the self-relationship doesn't fight the delete sweep (delete children
  before parent, or rely on the `.cascade` rule).
- `gatherLocalChanges`: include `parentId` on the `TodoChange` for creates; it
  is immutable so it needn't be re-sent on updates, but sending it is harmless
  (server ignores it on update).

`Views/Components/TodoEditSheet.swift`

- Add a **Subtasks** `Section` to the edit `Form` (alongside Notes / Due Date /
  Priority / Repeat):
  - List of child rows: checkbox + title. Active rows drag-reorderable
    (`.onMove`), completed rows pinned to the bottom and not movable.
  - "Add subtask" row/field that creates a `TodoItem` with `parentId` set and a
    position at the end of the active sibling group.
  - Ordering: active subtasks by `position` asc, then completed below.
- **Recurrence exclusivity**: hide the Repeat section when the todo has
  subtasks; hide the Subtasks section when the todo repeats.

`ViewModels/TodoViewModel.swift`, `Views/Components/TodoItemRow.swift`

- Exclude subtasks (`parentId != nil`) from the main list; only top-level rows
  render. Show an `n/m` progress indicator on parent rows.
- Parent checkbox cascades to subtasks (optimistic), mirroring the server;
  toggling a subtask never toggles the parent.
- Deleting a parent removes its subtasks (local `.cascade` + reconcile).

### Phase 5 — AI subtask generation (fast-follow, separate PR)

The on-brand payoff. Extend `smart-create` (and/or conversational refinement in
`reply.ts`) so the model can return a parent todo plus a list of subtasks —
"Plan a birthday party" → parent + 5 subtasks in one shot. Reuses the existing
tool-calling / structured-output plumbing; each subtask is inserted with
`parentId` set. Specced separately once Phases 1–4 land.

## Key considerations

- **Recurrence × subtasks**: disallow both on one todo (Phase 2 guards the
  cascade against the recurring-completion branch). Enforce **server-side**
  (`sync.ts` + web server functions) as the source of truth, and mirror it in
  the UI (hide "add subtask" on recurring todos and "make repeating" on todos
  with subtasks) so the illegal state can't be reached by an old client or a
  direct API call.
- **`parent_id` immutability** removes the hardest sync case (reparenting).
  Keep it immutable everywhere; do not add a "move to parent" action.
- **Completion `completedAt`**: for subtasks, `completedAt` is a plain
  completion timestamp (they can't recur), set when completed and cleared to
  null when reopened.
- **Lists**: list filters resolve subtask membership through the parent — a
  subtask shows in a list iff its parent is in that list. No `todo_lists` rows
  for subtasks.
- **Counts/badges**: top-level only (decision 10). When computing the app badge
  and list counts, filter `parentId == null` so subtasks never inflate the
  number.
- **Subtask UI surface**: subtasks are managed inside the expanded todo form,
  not as nested main-list rows (decision 9). A subtask row in that form is a
  checkbox + title only for v1; richer per-subtask fields (its own due date,
  notes) can be surfaced later without schema changes since a subtask is a full
  todo.

## Acceptance criteria

- [ ] `todos.parent_id` migration applied; existing todos unaffected
      (`parent_id` null).
- [ ] Creating a subtask on web and iOS persists `parentId` and syncs to the
      other platform.
- [ ] Subtasks are listed and managed inside the expanded todo form on web
      (`TodoItemExpanded`) and iOS (`TodoEditSheet`); the main-list parent row
      shows an `n/m` progress indicator and does not render subtasks as rows.
- [ ] Checking a parent completes all its subtasks; unchecking reopens them,
      on both platforms and after a round-trip sync.
- [ ] Completing all subtasks does **not** complete the parent.
- [ ] Completed subtasks sit at the bottom of the parent's list and are not
      reorderable; active subtasks reorder within the sibling group.
- [ ] Deleting a parent removes its subtasks on every device (cascade +
      reconcile), with no orphaned rows.
- [ ] A subtask cannot be promoted to top-level or moved to another parent.
- [ ] Recurrence and subtasks are mutually exclusive on a single todo,
      enforced server-side and reflected in the edit UI.
- [ ] The app badge and list counts include only top-level todos.
- [ ] `pnpm typecheck`, `pnpm lint`, `pnpm test` pass; iOS builds and
      SwiftLint passes.

## Resolved during brainstorm

- **Recurring parent with subtasks** → mutually exclusive for v1 (decision 7),
  enforced server-side + UI.
- **Badge/counts** → top-level only (decision 10).
- **Subtask UI** → a list inside the todo's expanded/edit view, not nested
  main-list rows (decision 9).
- **Parent delete** → cascades to children (decision 8).

## Open questions

- **Uncheck-a-child while parent is complete**: current design leaves the parent
  complete (parent completion is explicit; the cascade only fires on a parent
  toggle). Flagging in case a different feel is wanted — otherwise it ships as
  described.

## Future / out of scope

- `deletedIds`/tombstones to move iOS off always-full-pull to true incremental
  delta sync. Orthogonal optimisation; separate plan.
- Richer per-subtask editing (a subtask's own due date/notes) — no schema
  change needed since a subtask is already a full todo.
