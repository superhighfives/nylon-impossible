## Problem

Todos are currently flat. Real work is often hierarchical â€” "Plan a birthday
party" is really a handful of smaller steps. We want **subtasks**: todos that
belong to a parent todo, synced across web, iOS, and the API, and fitting the
app's existing "describe it, AI handles the rest" ethos (a subtask is a
first-class todo, so it inherits URLs, AI enrichment, research, and sync for
free).

## Decisions (agreed during brainstorm)

1. **One level only.** Parent â†’ subtasks, no grandchildren. A subtask cannot
   itself have subtasks.
2. **A subtask _is_ a todo.** Modelled as a self-referential `parent_id` on
   `todos`, not a separate lightweight table. Subtasks reuse the whole todo
   pipeline (notes, due date, priority, URLs, AI, research, sync).
3. **Completion is a downward master switch.**
   - Checking a parent completes **all** its subtasks; unchecking a parent
     reopens **all** its subtasks.
   - Completing every subtask does **not** auto-complete the parent (no upward
     propagation). The parent's completion is always an explicit user action.
   - Completed subtasks sink to the **bottom** of the parent's subtask list.
4. **Subtasks are permanently bound to their parent.** No promotion to
   top-level, no moving between parents. `parent_id` is immutable after
   creation. This significantly simplifies sync â€” there are no reparenting
   conflicts to resolve.
5. **Ordering.** Active subtasks are reorderable via fractional index within
   their sibling group. Completed subtasks are pinned to the bottom and are
   **not** reorderable.
6. **Lists.** Subtasks inherit their parent's list membership. They never
   appear independently in lists (no rows in `todo_lists` for subtasks; list
   filtering resolves through the parent).
7. **Recurrence is top-level only, mutually exclusive with subtasks.** A
   subtask cannot recur, and a todo may have subtasks **or** a recurrence, not
   both. Enforced server-side (source of truth) and mirrored in the UI.
8. **Deletes cascade; no new machinery needed.** Deleting a parent deletes its
   children. Both clients already reconcile deletions by absence (iOS always
   pulls with `lastSyncedAt: nil`; web refetches the full list), so D1's
   `ON DELETE cascade` propagates a parent deletion to its children on every
   device with no server changes. `deletedIds`/tombstones remain a worthwhile
   _future_ optimisation (true incremental delta sync) but are out of scope.
9. **Subtasks live inside the todo.** They are displayed and managed as a list
   *within* the todo's expanded/edit view (web `TodoItemExpanded`, iOS
   `TodoEditSheet`) â€” an "add subtask" input plus a reorderable list of child
   rows with checkboxes, completed rows pinned to the bottom. Subtasks are
   **not** rendered as nested rows in the main list; the main-list parent row
   shows only an `n/m` progress indicator.
10. **Badge/counts are top-level only.** The app icon badge and any "N todos"
    counts include only top-level todos, never subtasks, so a task with many
    subtasks never inflates the number.

## Solution

Add a nullable, self-referential `parent_id` to `todos`
(`references(() => todos.id, { onDelete: "cascade" })`). Top-level todos have
`parent_id IS NULL`; subtasks point at their parent. Positioning stays
fractional-index but is scoped to the sibling group `(user_id, parent_id)`.

The API keeps returning a **flat** list of todos (both `getTodos` and
`/todos/sync` already do this); each client reconstructs the one-level tree
from `parentId`. This keeps the API surface tiny.

## Implementation

### Phase 1 â€” Data model & shared schema

`src/shared/src/schema.ts`

- Add to the `todos` table:
  ```ts
  parentId: text("parent_id").references(
    (): AnySQLiteColumn => todos.id,
    { onDelete: "cascade" },
  ),
  ```
  (self-reference needs the `AnySQLiteColumn` return annotation; import from
  `drizzle-orm/sqlite-core`).
- Add an index for sibling lookups/ordering, scoped per user to match the
  `(user_id, parent_id)` sibling grouping (userId first so subtask queries stay
  per-user and don't scan across the whole table):
  ```ts
  index("idx_todos_user_parent_position").on(table.userId, table.parentId, table.position),
  ```
  Keep `idx_todos_user_position` for the top-level list.
- Extend `todosRelations` with a self relation:
  ```ts
  parent: one(todos, { fields: [todos.parentId], references: [todos.id], relationName: "subtasks" }),
  subtasks: many(todos, { relationName: "subtasks" }),
  ```

Migration: this repo **hand-writes** migration SQL (the drizzle meta journal
only tracks the initial `0000`/`0001`; `0002`â€“`0016` are hand-written and
`db:generate` diffs against a stale snapshot, so it isn't used). Add
`src/api/migrations/0017_add_parent_id_to_todos.sql`:
```sql
ALTER TABLE todos ADD COLUMN parent_id TEXT REFERENCES todos(id) ON DELETE CASCADE;
CREATE INDEX idx_todos_user_parent_position ON todos(user_id, parent_id, position);
```
Apply locally with `pnpm db:migrate` (`wrangler d1 migrations apply`, tracked in
the `d1_migrations` table).

### Phase 2 â€” API

`src/api/src/handlers/sync.ts`

- **`serializeTodo`**: add `parentId: todo.parentId?.toLowerCase() ?? null`.
- **`syncRequestSchema`**: add `parentId: z.string().uuid().nullable().optional()`
  on each change.
- **Insert path**: persist `parentId` on create (normalise to lowercase).
- **Update path**: **ignore** `parentId` on update â€” it is immutable (do not
  read it into the `.set(...)`). Reparenting is not a supported operation.
- **Completion cascade (parent â†’ children)**: when an update sets a top-level
  todo's `completed` to a new value, cascade it to its children in the same
  request:
  ```ts
  if (change.completed !== undefined && change.completed !== existing.completed && !existing.parentId) {
    await db.update(todos)
      .set({ completed: change.completed, completedAt: change.completed ? now : null, updatedAt: now })
      .where(eq(todos.parentId, normalizedId));
  }
  ```
  Bumping children's `updatedAt` guarantees they're picked up by the next pull
  on every client. (Guard against combining with the recurring-completion
  branch â€” a todo with subtasks is not allowed to recur.)
- **Delete path**: no change needed. D1 `ON DELETE cascade` removes children;
  clients reconcile by absence.
- **Recurrence Ă— subtasks invariant (server-side)**: enforce it here, not just
  in the UI, so older clients and direct API calls can't create an illegal
  state. Reject (or drop) a `recurrence` on a todo that has children, reject a
  `parentId` create under a recurring parent, and reject adding a subtask to a
  recurring todo. Mirror the same guard in the web server functions below.

`src/web/src/server/todos.ts`

- `getTodos` already returns a flat list â€” add `parentId` to whatever
  serialisation/mapping the web server function returns (mirror the sync
  serializer). No ordering change required; the client groups by `parentId`.
- `createTodo` / `updateTodo`: accept and persist `parentId` on create; ignore
  on update. Add the same parentâ†’children completion cascade on update, and the
  same server-side recurrence Ă— subtasks guard as `sync.ts`.
- `deleteTodo`: unchanged (cascade handles children).

`src/api/src/handlers/smart-create.ts`

- No change in Phase 2. (AI subtask generation is Phase 5.) A subtask created
  manually just posts a normal todo with `parentId` set and a position
  generated within the sibling group.

## Key considerations

- **Recurrence Ă— subtasks**: disallow both on one todo (Phase 2 guards the
  cascade against the recurring-completion branch). Enforce **server-side**
  (`sync.ts` + web server functions) as the source of truth, and mirror it in
  the UI (hide "add subtask" on recurring todos and "make repeating" on todos
  with subtasks) so the illegal state can't be reached by an old client or a
  direct API call.
- **`parent_id` immutability** removes the hardest sync case (reparenting).
  Keep it immutable everywhere; do not add a "move to parent" action.
- **Completion `completedAt`**: for subtasks, `completedAt` is a plain
  completion timestamp (they can't recur), set when completed and cleared to
  null when reopened.
- **Lists**: list filters resolve subtask membership through the parent â€” a
  subtask shows in a list iff its parent is in that list. No `todo_lists` rows
  for subtasks.
- **Counts/badges**: top-level only (decision 10). When computing the app badge
  and list counts, filter `parentId == null` so subtasks never inflate the
  number.
- **Subtask UI surface**: subtasks are managed inside the expanded todo form,
  not as nested main-list rows (decision 9). A subtask row in that form is a
  checkbox + title only for v1; richer per-subtask fields (its own due date,
  notes) can be surfaced later without schema changes since a subtask is a full
  todo.

## Overview

Subtasks add one level of todo hierarchy: a todo can have child todos ("Book the venue" under "Plan a birthday party"), synced across the API, web, and iOS. A subtask is modelled as a full todo (not a lightweight table), so it inherits URLs, notes, due dates, priority, AI enrichment, research, and sync for free. The design is deliberately minimal to keep sync trivial: one level only, `parent_id` is immutable (no reparenting â†’ no reparenting conflicts), completion is a downward-only master switch (checking a parent completes its children; unchecking reopens them; completing all children never auto-completes the parent), completed subtasks sink to the bottom, and recurrence and subtasks are mutually exclusive on a single todo. The API keeps returning a flat todo list; each client reconstructs the one-level tree from `parentId`. Deletes need no new machinery â€” D1's `ON DELETE CASCADE` plus the clients' existing reconcile-by-absence propagate a parent deletion to its children. The feature also ships AI-generated subtasks: the enrichment model can break a multi-step project into 2â€“6 action-oriented steps in one shot.

## Architecture

Data model (`src/shared/src/schema.ts`, migration `src/api/migrations/0017_add_parent_id_to_todos.sql`): a nullable self-referential `todos.parent_id TEXT REFERENCES todos(id) ON DELETE CASCADE`, plus `idx_todos_user_parent_position` on `(user_id, parent_id, position)` for sibling lookups/ordering (the older `idx_todos_user_position` stays for the top-level list). The Drizzle `todosRelations` gained `parent`/`subtasks` self-relations (`relationName: "subtasks"`). The migration is hand-written to match repo convention (drizzle's meta journal is stale past `0001`). Positioning is still fractional-index but scoped to the sibling group `(user_id, parent_id)`; top-level position queries in `getTodos`/smart-create were narrowed to `parent_id IS NULL` so subtasks don't interleave with top-level ordering.

API (`src/api/src/handlers/sync.ts`, `src/web/src/server/todos.ts`, `src/api/src/handlers/todos.ts`): `parentId` is carried through every read serializer (normalised to lowercase), persisted on create, and ignored on update (immutable). All three write paths share two behaviours: (1) a completion cascade â€” toggling a top-level todo's `completed` flag updates all rows where `parent_id = <id>`, bumping their `updatedAt` so the next pull picks them up; subtasks never stamp `completedAt` (they can't recur); and (2) the recurrenceĂ—subtasks guard, enforced server-side â€” a subtask never recurs, setting a recurrence on a todo that has children drops it, and adding a subtask to a recurring parent clears the parent's recurrence (an `isNotNull` guard avoids needless writes). `src/api/src/lib/db.ts` re-exports `isNull`/`isNotNull` for these queries. Coverage: 7 new sync integration tests.

Web (`src/web/src/components/SubtaskSection.tsx` â€” a new ~270-line component â€” inside `TodoItemExpanded.tsx`): add-subtask input, checkboxes, dnd-kit drag-reorder for active subtasks, completed pinned to the bottom (not reorderable), `n/m` count. `TodoList.tsx` renders top-level todos only (grouped by parent) with an `n/m` progress badge on parent rows; the parent-checkbox cascade is optimistic in `useTodos.ts` (`useUpdateTodo`) mirroring the server. Repeat control and Subtasks section hide each other. 6 new component tests (147 web tests green).

iOS (`Models/TodoItem.swift`, `Services/APIService.swift`, `Services/SyncService.swift`, `Services/BackgroundSyncService.swift`, `Services/TaskCreationService.swift`, `ViewModels/TodoViewModel.swift`, `Views/Components/TodoEditSheet.swift`, `Views/Components/TodoItemRow.swift`, `ContentView.swift`): `parentId` flows through the `APITodo`/`TodoChange` DTOs and both `applySync` branches + `gatherLocalChanges`. A Subtasks section in `TodoEditSheet` mirrors the web UX (add field, tap-to-toggle, swipe-delete, `.onMove` reorder for active, completed pinned below, `n/m` header); the main list shows top-level only with an `n/m` badge; the parent cascade lives in `TodoViewModel.toggleTodo`.

**Deviations from the plan:**
- **iOS dropped the SwiftData `@Relationship`.** The plan proposed a self-referential `@Relationship(deleteRule: .cascade)` with `subtasks`/`parent` on `TodoItem`. Shipped code uses a plain `var parentId: UUID?` with no relationship â€” grouping and cascade are handled explicitly. This was a deliberate de-risking choice because the change was written without a local Swift toolchain (CI runs SwiftLint only), avoiding self-referential-relationship pitfalls in un-compiled code.
- **iOS is not build-verified.** As flagged in the plan and PR, it needs an Xcode build + simulator pass. A compile bug (a missing `return` in a `TodoChange.orderedForSync()` extension added during review) had to be fixed in a follow-up commit, confirming the un-compiled risk was real.
- **Phase 5 (AI subtask generation) shipped in this same PR, not as a separate fast-follow.** The plan explicitly deferred it to "a separate PR"; it landed in #218 instead.
- **AI generation was built into the enrichment path, not `smart-create`/`reply.ts` as Phase 5 originally sketched.** It lives in `src/api/src/lib/ai.ts` (a `subtasks` property on `enrichTodoTool` / `TodoEnrichment`, with prompt guidance, examples, and sanitisation capping at 6) and `src/api/src/lib/ai-enrich.ts` (`enrichOrAskWithAI` inserts the child todos with `parentId` set and sibling-scoped positions, skips insertion if the todo already has children so re-enrichment can't duplicate, and drops recurrence when subtasks win). 4 new AI unit tests (294 API tests green).

The one open question the plan raised â€” unchecking a subtask while the parent is complete leaves the parent complete â€” shipped as designed (parent completion is always explicit).
